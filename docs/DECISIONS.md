# 决策记录

## 2026-06-16 - Phase 01 技术基线

背景：

Phase 01 需要建立第一套可执行的技术基线、结构基线和设计基线，并在此基础上实现“前端团队总枢纽首页”。当前仓库没有前端框架、`package.json`、页面代码或部署配置。

决定：

- 技术栈采用 Astro + React Islands + TypeScript。
- 当前具体版本固定为 Astro 4 + React 18，以兼容本地 Node `18.19.1`。
- 包管理器采用 `pnpm`。
- 样式方案以 Astro scoped CSS 为主，复杂组件可使用 CSS Modules。
- Phase 01 先做本地验收，不配置生产部署。
- Phase 01 不建设数据库、后台、登录权限、自动采集或报告生成。

原因：

- 总枢纽首页以内容展示、结构说明和系统入口为主，适合静态优先。
- Astro 能保持首阶段实现轻量，同时为后续站内路由、局部交互和业务模块扩展保留空间。
- TypeScript 有利于后续内容配置、入口配置和组件边界维护。

影响：

- 第一阶段将创建 Astro 项目结构和基础脚本。
- 后续业务模块可逐步从静态入口、占位页或外链过渡为站内交互模块。
- 如果未来短期内需要复杂后台、强权限和服务端能力，需要重新评估是否继续沿用 Astro 或引入后端服务。
- 若未来升级 Node 到 Astro 5/6 要求的版本，可以再评估是否升级 Astro 主版本。

## 2026-06-16 - Phase 01 首页内容来源

背景：

`docs/references/Hero_Ref.md` 是总枢纽首页内容与结构参考，但用户明确说明该文件只用于说明页面应包含什么内容，不是样式参考，也不要求复刻 Markdown 格式。

决定：

- Phase 01 首页内容以 `docs/references/Hero_Ref.md` 为主要内容范围来源。
- 实现时应将其中内容拆为可维护的 TypeScript 配置。
- 不复刻 `Hero_Ref.md` 的 Markdown 标题层级、段落格式或列表样式。
- 首页视觉以 `docs/DESIGN.md` 的系统设计原则为准，并在实现中形成适合网站连续阅读的结构。

原因：

- Markdown 文件适合表达内容范围，但不等同于最终信息架构、视觉层级或网页布局。
- 首页应是连续纵向网站，不是 Markdown 渲染页，也不是 PDF/PPT 复刻。

影响：

- 首页实现需要重新组织信息层级、版式、组件和响应式布局。
- 后续修改内容时优先维护配置，而不是在页面组件中散落硬编码。

## 2026-06-16 - 总枢纽首页边界

背景：

产品定义明确总枢纽首页是系统入口，不是整个前端团队管理系统，也不是业务驾驶舱。

决定：

- 首页不展示实时数据、周报明细、月报明细、当前项目列表、业务排期表或个人绩效薪资明细。
- 首页提供业务系统、组织与人员、绩效与报酬三个方向的入口。
- Phase 01 入口支持外链、占位和“建设中”状态，不实现登录权限。

原因：

- 避免总枢纽首页成为不可维护的信息堆叠页。
- 保持业务系统、组织人员系统和绩效报酬系统的概念与权限边界。

影响：

- 首页组件应服务解释、导航和入口。
- 实时业务内容应留到未来业务模块中实现。
