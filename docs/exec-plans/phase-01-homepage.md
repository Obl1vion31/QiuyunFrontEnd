## 2. 当前仓库状态

当前状态：

- 已有 Git 仓库，`main` 与 `origin/main` 同步。
- 没有 `package.json`。
- 没有 Vite、Next.js、Astro 或其他前端框架配置。
- 没有页面、组件、样式、测试或构建代码。
- 没有部署配置。
- `docs/DECISIONS.md` 和 `docs/ROADMAP.md` 为空，尚无正式技术决策。
- `docs/PRODUCT.md`、`docs/STRUCTURE.md`、`docs/DESIGN.md`、`docs/references/Hero_Ref.md` 已足够进入 Phase 01 计划制定，但还不足以直接开始实施。

已经确定：

- 总枢纽首页是系统入口，不是整个系统。
- 首页不承载周报、月报、实时数据、当前项目列表或完整业务排期。
- Phase 01 可以静态优先，但不能阻碍未来业务模块扩展。
- 外部智能文档可以作为过渡入口。
- 中文阅读体验和信息层级非常重要。

已批准选择：

- 技术栈：Astro + React Islands + TypeScript；
- 包管理器：`pnpm`；
- 样式方案：Astro scoped CSS 为主，复杂组件可使用 CSS Modules；
- 内容配置方式：首页内容拆为 TypeScript 配置；
- 路由方式：Phase 01 先实现首页路由，未来模块再扩展站内路由；
- 验收方式：Phase 01 先做本地验收，不配置生产部署；
- `Hero_Ref.md` 使用方式：只作为内容范围，不作为样式或 Markdown 格式参考。

## 3. 技术方案比较与推荐

### 方案比较

| 方案 | 优点 | 风险 | 适配度 |
| --- | --- | --- | --- |
| Astro + React Islands | 静态首页非常适合；Markdown 和内容维护友好；未来可逐步加入局部交互；首阶段不容易过度设计 | 如果短期内要发展成复杂后台，需要额外明确服务端和后台扩展方案 | 高 |
| Next.js App Router | 未来数据库、权限、报告、后台扩展路径完整；生态成熟 | Phase 01 只做总枢纽首页时偏重，容易提前引入复杂度 | 中高 |
| Vite + React | 简单、快速、Codex 易维护；适合单页静态实现 | 后续路由、内容、服务端能力、报告生成和权限体系需要自行补架构 | 中 |

### 推荐方案

推荐采用：

- Astro；
- React Islands；
- TypeScript；
- Astro scoped CSS 为主；
- 复杂组件可使用 CSS Modules；
- `pnpm` 作为包管理器。

### 推荐理由

- Phase 01 首页以内容展示、结构说明和入口导航为主，Astro 的静态优先模型最贴合。
- 后续业务录入、内容排期、项目管理可以通过站内路由、React 交互岛或后续后端服务逐步扩展。
- 中文内容和配置可以用 Markdown、TypeScript 配置或 JSON 数据集中维护。
- 总枢纽首页可以生成静态页面，部署、预览和回滚简单。
- 不需要在第一阶段提前引入数据库、权限、SSR 或完整后台。
- Codex 后续维护时，内容、组件和页面边界较清楚。

### Phase 01 使用的能力

- Astro 页面与静态构建；
- TypeScript 内容配置；
- React 只用于需要交互的局部组件；
- Astro scoped CSS 或 CSS Modules；
- 静态系统入口配置；
- 本地构建和响应式检查。

### Phase 01 暂不启用的能力

- 数据库；
- 登录和权限；
- 后台管理；
- API 路由；
- SSR；
- 自动报告生成；
- 爬虫或第三方数据同步；
- 生产部署自动化。

### 未来扩展路径

- 业务模块可从静态占位页逐步升级为 React 交互页面。
- 系统入口配置可从本地配置迁移到 CMS、数据库或后台。
- 周报/月报可先做静态报告页，后续接结构化数据和 PDF 生成。
- 权限、数据库、后台管理可在正式进入业务系统阶段再决策。

### 主要风险

- 如果短期内确定必须建设复杂后台或强权限系统，Next.js 可能更直接。
- Astro + React Islands 需要明确“静态内容”和“交互模块”的边界，避免后续混乱。
- 首页视觉如果大量依赖复杂动效，需要控制实现成本和可维护性。

### 已获得用户确认

- 采用 Astro + React Islands。
- 使用 TypeScript。
- 采用 `pnpm`。
- 采用 Astro scoped CSS 为主、CSS Modules 为辅。
- 将 `Hero_Ref.md` 中的首页内容拆成可维护配置。
- Phase 01 先本地验收，预览环境另行决策。
- `Hero_Ref.md` 只作为内容参考，不作为样式参考。

### 阶段 3：结构基线

Phase 01 中更新 `docs/STRUCTURE.md`，但不把它写成完整系统蓝图。

现在可以确定：

- 总枢纽是入口，不是业务驾驶舱。
- 首页连接业务系统、组织与人员、绩效与报酬三个方向。
- 业务数据、周报、排期、风险和项目明细不放在首页。
- 外部智能文档可作为过渡入口。
- 系统入口需要配置化，未来可从外链替换为站内路由。

需要后续确认：

- 第一版是否只做首页，还是包含占位路由。
- 三个系统入口的真实链接、状态和权限提示。
- 未来业务模块的路由分组方式。
- 移动端导航是否需要页内锚点。

必须留到未来业务模块再决定：

- 业务数据字段；

- 周报/月报生成规则；

### 阶段 4：设计基线

Phase 01 中更新 `docs/DESIGN.md`，只记录长期设计原则，不建立完整设计系统。

可以在 Phase 01 确认：

- 中文阅读体验优先；
- 首页是连续纵向网站，不是 PPT；
- 总枢纽视觉克制、安静、有方向感；
- 深色 Hero、暖色强调和清晰层级可作为首页方向；
- 业务系统和数据页面未来应共享同一套视觉语言，但信息密度不同；
- 卡片只用于明确的模块、入口或信息组，不做过度装饰。

首页实施中验证：

- Hero 视觉是否既接近 `Hero_Ref.md` 又适合网页；
- 协作链路在桌面端和移动端是否可读；
- 三个系统入口是否清楚；
- 页面是否避免首屏信息过载；
- 中文字号、行距、留白是否自然。

属于整个系统的规则：

- 中文层级；
- 响应式原则；
- 导航与入口一致性；
- 视觉克制；
- 数据、报告、业务操作页面的可扫描性。

只属于总枢纽首页的规则：

- 深色 Hero 场景；
- `FRONTEND TEAM HUB` 辅助标题；
- 暖色强调核心表达；
- 连续纵向转化 PDF 四页内容；
- 首页维护规则展示。

### 阶段 6：首页实施

进入实施前已确认：

- Phase 01 计划；
- 技术选型；
- 结构方向；
- 设计方向；
- 需要写入 `docs/DECISIONS.md` 的正式决定。

实施步骤：

1. 初始化技术框架  
   创建 Astro + TypeScript 项目结构，配置基础脚本和目录。

2. 建立内容配置  
   将 `Hero_Ref.md` 中的首页内容转为集中配置，包括 Hero、使命、目标、链路、执行底座、岗位、系统入口、使用路径和更新时间。

3. 建立页面结构  
   实现连续纵向首页，不做轮播、不做四页 PPT。

4. 建立组件边界  
   公共组件只保留真正可复用的布局、入口卡片和区块标题；首页专用组件留在首页范围内。

5. 实现协作链路  
   桌面端使用可读的纵向流程轴；移动端保持单列，不压缩成横向流程图。

6. 实现系统入口  
   三个入口配置化，支持外链、占位或建设中状态；绩效与报酬入口体现授权访问，但不实现登录。

7. 响应式与视觉完善  
   桌面端、平板和移动端均可读；Hero 不信息过载；中文层级自然。

8. 构建和测试  
   运行格式检查、类型检查、构建、本地预览、移动端和桌面端人工验收。

9. 本地验收  
   提供本地访问地址和验收说明。

10. 预览环境  
   仅在用户批准部署方式后配置，不在 Phase 01 技术计划批准前执行。

11. 最终提交  
   完成验收后再提交 Git；不自动推送，除非用户要求。

首页完成标准：

- 用户能理解前端团队为什么存在。
- 用户能看懂从规划到咨询再到反馈回流的完整链路。
- 用户能看出每个环节主要负责人。
- 用户能找到业务、组织与人员、绩效与报酬三个入口。
- 首页不出现实时业务数据、周报明细或项目明细。
- 页面是连续纵向网站。
- 桌面端和移动端均可正常阅读。
- 系统入口未来可从外链替换为站内路由。
- 内容和组件结构便于维护。
- 不引入数据库、后台和权限系统。
