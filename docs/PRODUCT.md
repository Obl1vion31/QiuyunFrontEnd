# 产品定义

## 1. 产品名称

暂定名称：前端团队管理系统。

可在后续讨论的命名方向：

- 前端业务管理系统
- 前端运营中台
- 前端团队运营系统
- 前端业务总枢纽
- 前端协作与复盘系统

在团队确认产品范围和命名方向前，不应擅自更改暂定名称。

## 2. 产品背景

当前团队依赖智能文档和人工流程管理内容运营、内容策划、视觉设计需求、推广帖和非推广帖、用户触点、销售线索交接、数据复盘、周例会、月例会以及内部方法沉淀。

文档化方式在早期具备灵活性，但长期会带来一些问题：

- 关键信息容易分散在多个文档中；
- 同一份数据可能被复制到不同汇总、报告或视图中；
- 周报和月报需要反复人工整理；
- 历史问题、决策和跟进结果难以持续追踪；
- 业务运营、规则、模板、人员信息和管理输出容易混在一起；
- 不同角色需要不同视图，但文档通常只能呈现一套共享结构。

因此，系统需要从“文档优先”逐步走向“网站化产品结构”。智能文档仍可作为过渡或辅助工具继续存在，但网站应成为更清晰的入口，用于承载导航、业务结构、未来模块边界和可复用的管理输出。

## 3. 产品使命

前端团队管理系统应帮助前端业务团队以可复用、可扩展的方式管理工作、数据、规则、复盘输出和协作路径。

长期使命包括：

- 说明前端团队为什么存在，以及如何运作；
- 连接计划、执行、数据、复盘和决策跟进；
- 减少同一信息的重复人工维护；
- 区分原始数据、业务视图、管理报告和组织材料；
- 为不同用户提供合适的入口和视图；
- 在第一阶段不强行建设完整后台的前提下，为未来业务模块保留扩展空间。

## 4. 目标用户

### 前端主管

需要清楚了解团队运作、业务进展、复盘输出、风险、决策和后续跟进状态。

### 新媒体运营

需要管理内容计划、发布排期、推广帖和非推广帖、账号数据、帖子表现和复盘材料。

### 视觉设计

需要明确设计需求、优先级、截止时间、相关内容背景、反馈记录，以及可复用的设计或内容模板。

### 用户触点运营

需要了解用户来源、咨询记录、触点承接、线索交接状态和后续跟进要求。

### 相关协作人员

可能包括销售、项目协作方、管理层或其他需要查看部分前端信息、接收结构化交接结果的团队。

### 未来系统管理员

如果产品未来从静态优先阶段发展为真正的内部系统，可能需要有人管理系统配置、模块可见性、角色权限、数据规则和集成设置。

## 5. 核心问题

- 内容计划、运营数据、会议结论、规则和模板可能分散在多个文档中。
- 数据可能在不同汇总、报告或视图中被重复维护。
- 周报和月报容易变成人工输出，而不是由数据生成的管理视图。
- 历史问题和决策难以从发现、处理到结果复查形成连续追踪。
- 业务数据、SOP、人员信息、绩效内容和报酬内容需要更清晰的概念和权限边界。
- 总枢纽首页可以解释团队和系统，但不能承载所有实时业务信息。
- 主管和员工需要不同层级的汇总、细节和行动信息。
- 系统需要支持未来模块扩展，但第一阶段不应因此直接建设完整后台、数据库或权限系统。

## 6. 产品能力地图

以下地图只描述可能存在的高层模块，不定义最终字段、数据模型或详细功能。

| 模块 | 当前状态 | 高层用途 |
| --- | --- | --- |
| 总枢纽 | 已确认 | 解释团队使命、工作链路、岗位协作、系统入口和用户路径。 |
| 业务计划与内容排期 | 可能需要 | 管理月度计划、选题计划、发布排期、推广/非推广安排、负责人、时间、状态和进度。 |
| 项目和任务 | 可能需要 | 跟踪临时项目、周期任务、负责人、截止时间、执行状态和未解决事项。 |
| 推广与非推广数据 | 可能需要 | 管理不同内容类型的数据表现，并避免在报告中重复维护同一数据。 |
| 用户触点 | 可能需要 | 跟踪用户咨询、线索来源、来源帖子、交接状态和触点过程数据。 |
| 周报与月报 | 可能需要 | 基于已维护数据生成主管版周报和月报，包括趋势、风险、结论和待拍板事项。 |
| 数据复盘与报告 | 可能需要 | 支持帖子复盘、项目复盘、内容类型对比、历史对比、结论生成和报告导出。 |
| 问题和决策跟踪 | 可能需要 | 跟踪未解决问题、风险、决策、负责人、后续状态和结果复查。 |
| SOP、规则与模板 | 可能需要 | 沉淀业务方法、内容模板、设计需求模板、数据录入规则、复盘模板和报告生成规则。 |
| 组织与人员 | 尚待判断 | 在适合纳入本系统时，管理人员结构、岗位、培训、职级和晋升路径。 |
| 绩效与报酬 | 尚待判断 | 仅在权限、隐私和架构边界清晰时，管理敏感绩效和报酬信息。 |
| 系统设置与权限 | 尚待判断 | 如果产品发展为真正的内部系统，用于配置模块、角色、权限、数据来源和集成。 |

## 7. 产品边界

### 前端系统与销售系统的边界

前端系统应管理前端侧的计划、内容、触点、来源数据、复盘输出和交接记录。

它不应替代销售系统。销售跟进、销售管线、成交管理和最终销售业绩应继续属于销售侧系统，除非后续明确建立集成边界。

### 网站与智能文档的边界

网站应成为结构化入口和产品框架。

智能文档可以继续承载过渡材料、原始记录、长篇 SOP，或暂时变化过快、不适合立即产品化的内容。

第一阶段网站可以跳转到智能文档，但长期会被多个视图复用的数据，应尽量只在一个结构化来源中维护。

### 原始数据与报告输出的边界

原始业务数据不应主要存在于周报或月报中。

周报和月报应是基于已维护数据、计算结果、结论和决策记录生成的管理输出。

这个边界很重要：报告是视图，不是事实来源。

### 当前阶段与未来阶段的边界

当前阶段应明确产品结构、信息架构和可扩展的第一界面。

当前阶段不应直接完成完整后台、数据库、权限模型、数据自动化、报告引擎、人员模块或报酬模块。

## 8. 第一阶段目标

第一阶段应聚焦建立稳定基础，而不是完成整个系统。

建议目标：

- 建立产品定义和模块边界；
- 建立初始信息架构；
- 在产品结构批准后，再建立基础技术框架；
- 在产品结构批准后，再实现总枢纽作为第一个用户界面；
- 为未来业务、组织和绩效相关模块提供可配置入口；
- 保持智能文档作为过渡外部参考或链接资源；
- 避免设计和代码决策阻碍未来模块扩展。

第一阶段可以静态优先，但概念上不应被限制为静态宣传网站。

## 9. 第一阶段非目标

第一阶段不应：

- 建设完整业务后台；
- 建设完整数据库；
- 建设复杂登录和权限系统；
- 自动采集所有平台数据；
- 一次性迁移所有智能文档；
- 完成全部业务模块；
- 在边界尚未验证前建设组织、绩效和报酬模块；
- 将总枢纽首页当作实时数据驾驶舱；
- 将本周结论、实时风险或临时项目列表直接堆放在总枢纽首页。

## 10. 已确认方向、合理假设与开放问题

### 已确认方向

- 这是一个长期可更新、可复用、可扩展的前端业务系统。
- 系统不能最终停留在静态展示页。
- 第一阶段可以静态优先，但需要保留未来扩展边界。
- 总枢纽是系统入口，不是整个系统。
- 当前周度结论、实时数据、项目状态和当前风险属于业务模块，不直接堆放在总枢纽首页。
- 数据应尽量只维护一次，并复用于不同视图和报告。
- 周报和月报是管理输出，不是原始数据存储位置。
- 业务系统、组织人员系统、绩效报酬系统需要保持概念和权限边界。
- 第一阶段不应直接建设完整后台、数据库或复杂权限系统。
- 外部智能文档可以继续作为过渡路径。
- 桌面端和移动端阅读体验都需要考虑。
- 中文信息层级和可读性非常重要。

### 合理假设

- 总枢纽很可能是产品结构批准后的第一个实现界面。
- 业务计划、排期、数据复盘、报告和问题跟踪可能成为早期或中期模块。
- 组织、绩效和报酬模块可能需要更严格的权限，不一定与业务模块使用同一实施节奏。
- 第一阶段的部分系统入口可能先链接到智能文档或占位模块页面。
- 数据自动化可以后置；过渡阶段可以接受人工录入或文档来源。

### 开放问题

- 第一版网站必须包含哪些模块，哪些继续保留在外部智能文档中？
- 总枢纽的最小可用版本应包含什么？
- 信息架构应如何区分业务运营、报告、SOP、人员和绩效内容？
- 哪些数据应优先人工录入，哪些数据未来可能同步或自动采集？
- 第一个可用版本是否需要登录，还是可以先在网站外部处理访问控制？
- 哪些数据应从文档内容转为结构化数据源？
- 周报和月报最终应以网页、PDF，还是两者同时存在？
- 哪些用户需要主管级视图，哪些用户需要岗位工作视图？
- 人员、绩效和报酬是否应放在同一个产品壳中，还是继续作为独立系统或文档？
- 在产品结构批准后，应选择什么部署环境和技术栈？
