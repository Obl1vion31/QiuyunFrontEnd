# Codex 网站建设使用说明

当你希望 Codex 帮你规划、设计、开发、调试或优化网站和 Web 应用时，可以参考这份说明。

## 最好的开始方式

如果是一个新网站，优先给出清晰的产品简述，而不是只说一个模糊需求。

推荐起始提示词：

```text
为【目标用户】建设一个网站，帮助用户完成【核心目标】。
视觉风格希望是【风格方向】。
核心页面或功能包括：
- 【页面/功能 1】
- 【页面/功能 2】
- 【页面/功能 3】
如果项目已有技术栈，请沿用现有技术栈。如果当前文件夹为空，请选择一个实用的现代前端方案。
完成实现后，请本地运行并告诉我访问地址。
```

如果你还在探索想法，先让 Codex 做计划：

```text
先不要写代码。请帮我明确网站结构、页面范围、视觉方向和 MVP 功能。
等我确认后再进入实现。
```

## 什么时候使用 `/init`

当 Codex 进入一个新的或不熟悉的仓库，并且需要先理解项目再修改时，适合使用 `/init`。

适合场景：

- 仓库中已经有代码；
- 项目有自定义规范；
- 需要 Codex 先检查脚本、依赖、路由、组件和样式方案；
- 需要长期围绕同一个项目持续协作。

示例：

```text
/init
请检查这个前端项目，并总结框架、开发命令、目录结构、样式方案，以及之后我应该如何让你修改它。
```

如果当前文件夹是空的，`/init` 不一定必要，可以直接让 Codex 搭建项目。

## 什么时候使用 Skills

Skills 是专门的能力说明包。当任务明显匹配某个 skill 的用途时，可以主动指定使用。

### `imagegen`

当网站需要自定义位图素材时使用，例如图片、产品图、纹理、背景、精灵图、视觉 mockup 或透明背景素材。

适合用于：

- Hero 背景图；
- 产品展示图；
- 游戏精灵图或视觉资产；
- 自定义插画；
- 修改或转换上传的图片。

示例：

```text
使用 imagegen skill，为一个咖啡订阅落地页生成高级感 Hero 背景图，然后围绕这张图建设页面。
```

如果素材应该用 SVG、CSS、HTML 或项目已有图标系统完成，就不要使用 `imagegen`。

### `openai-docs`

当网站涉及 OpenAI API、ChatGPT、Codex、模型、agents、图像生成、embeddings 或其他 OpenAI 产品行为时使用。

适合用于：

- 选择最新 OpenAI 模型；
- 实现 API 调用；
- 查询当前官方文档；
- 更新旧版 OpenAI 代码；
- 理解 Codex 或 OpenAI 工具能力。

示例：

```text
使用 openai-docs skill。我想给这个网站增加 OpenAI 聊天功能，请先检查当前官方文档，再进行实现。
```

### `canva:*` Skills

当网站工作依赖 Canva 素材、演示文稿、品牌套件或设计文件时使用 Canva 相关 skills。

适合用于：

- 将品牌演示文稿转化为网站内容；
- 翻译 Canva 设计；
- 为社交平台批量调整 Canva 营销素材尺寸；
- 创建品牌化 Canva 演示文稿。

示例：

```text
使用 Canva presentation skill，先把我的品牌简报转成演示文稿，然后我们再把这份演示文稿作为网站视觉参考。
```

## 什么时候适合快速处理

如果你的环境支持 `/fast` 或类似模式，并且速度比深度分析更重要，可以使用它。

适合：

- 小型文案修改；
- 简单 bug 修复；
- 样式微调；
- 增加一个组件；
- 快速原型。

不适合：

- 架构决策；
- 复杂重构；
- 登录、支付或安全敏感功能；
- 大范围 UI 重建；
- 原因不明确的生产问题排查。

示例：

```text
/fast
让导航栏支持移动端，并保持现有设计风格。
```

## 什么时候先要计划

当需求有多种可能方向时，先让 Codex 制定计划。

适合：

- 全新产品；
- 复杂后台或仪表盘；
- 多页面网站；
- 管理系统；
- 预约、交易或账号系统；
- 产品需求还不清晰的情况。

示例：

```text
先做计划。我想建设一个用于管理餐厅预约的 SaaS 仪表盘。
请定义页面、数据模型、组件和实施步骤。
在我批准计划前不要写代码。
```

## 什么时候可以直接让 Codex 实现

当目标明确、歧义较小时，可以直接要求实现。

示例：

```text
给首页增加一个价格区域，并支持月付/年付切换。
```

```text
修复结账页面的移动端布局问题。
```

```text
创建一个可复用的用户评价卡片组件，并用在落地页中。
```

## 常用提示词模式

### 建设新网站

```text
建设一个【网站类型】，面向【目标用户】。
主要目标：【目标】。
视觉风格：【风格】。
必需页面或区块：【列表】。
请使用真实感内容，并做好响应式。
完成后启动本地开发服务器。
```

### 改进现有网站

```text
请审查当前网站实现，然后在不改变产品范围的前提下改进 UX 和视觉设计。
保持现有技术栈和项目模式。
修改后运行检查，并启动本地开发服务器。
```

### 排查问题

```text
这个页面有一个问题：【描述问题】。
请在本地复现，找出根因，修复它，并说明改了什么。
```

### 安全重构

```text
重构【范围】，让它更容易维护。
保持行为不变。
如果项目已有测试配置，请增加或更新测试。
```

### 使用参考风格

```text
以【网站/应用/风格】作为视觉参考，但不要直接复制。
请把同等级的打磨、间距和交互质量应用到本项目。
```

## Codex 需要你提供的信息

为了获得更好的结果，尽量提供：

- 网站目的；
- 目标用户；
- 期望语气和视觉风格；
- 必需页面；
- 必需功能；
- 品牌颜色、Logo 或参考资料；
- 内容来源，或允许 Codex 编写真实感占位内容；
- 技术约束；
- 你希望快速原型、精细 MVP，还是生产级实现。

## 推荐网站需求模板

```text
我想建设一个网站：
【业务/产品/项目】

目标用户：
【目标用户】

主要用户目标：
【用户应该完成什么】

页面：
- 首页
- 【页面 2】
- 【页面 3】

功能：
- 【功能 1】
- 【功能 2】
- 【功能 3】

视觉方向：
【极简、高级、活泼、编辑感、SaaS、奢华、技术感等】

素材：
【Logo/图片/Canva 文件/暂无】

实现偏好：
【快速原型 / 精细 MVP / 生产级】

请检查仓库，选择合适的实现路径，完成建设，运行检查，并启动本地开发服务器。
```

## Codex 擅长做什么

Codex 在你允许它完成完整实施闭环时最有用：

- 检查现有代码库；
- 选择符合项目现状的实现方案；
- 直接修改文件；
- 构建组件和页面；
- 增加样式和响应式行为；
- 在合适时生成或接入素材；
- 运行格式化、测试或构建检查；
- 启动本地开发服务器；
- 总结实际修改内容。

## 应避免的提示词

尽量不要只说：

```text
做一个酷一点的网站。
```

```text
优化一下。
```

```text
让它现代一点。
```

这些表达过于模糊。应补充目标用户、网站目的、页面范围和风格方向。

## 推荐网站项目工作流

1. 如果仓库已有代码，先使用 `/init`。
2. 如果产品方向还不清楚，先要求制定计划。
3. 审核并调整计划。
4. 让 Codex 实现第一个完整版本。
5. 本地查看网站。
6. 提出有针对性的优化要求。
7. 让 Codex 运行检查并清理问题。
8. 继续推进功能、内容、SEO、可访问性和部署准备。

## 高质量后续请求示例

第一个版本存在后，使用更精确的请求：

```text
在不改变布局结构的前提下，让首页更有高级感。
```

```text
给价格切换增加键盘可访问交互。
```

```text
改进移动端导航，并在 375px 宽度下检查。
```

```text
把占位文案替换成面向小企业主的直接转化型文案。
```

```text
运行构建，修复所有错误，并告诉我还剩哪些风险。
```

## 简短原则

如果你想要速度，给 Codex 一个具体任务。

如果你想要质量，给 Codex 产品上下文。

如果你想要最好的结果，两者都要给。
