# AGENTS.md

本文件只记录 agent 工作入口规则：先看什么、如何判断边界、哪些事不能直接做。

完整产品定义、结构、设计、路线图和决策，不应重复写在本文件中。

## 1. 工作原则

- 先读相关文档，再行动。
- 只做用户明确要求或已经批准的任务。
- 未确认内容标记为“假设”或“开放问题”。
- 不把总枢纽首页当作整个系统。
- 不为尚未确认的未来能力过度设计。
- 如果用户要求“只制定计划”，完成计划后停止。

## 2. 必读路径

根据任务类型阅读对应文档：

- 产品范围、用户、模块边界：`docs/PRODUCT.md`
- 页面、导航、模块关系、用户路径：`docs/STRUCTURE.md`
- 视觉、交互、中文阅读体验：`docs/DESIGN.md`
- 阶段规划和优先级：`docs/ROADMAP.md`
- 已批准的重要决定：`docs/DECISIONS.md`
- 具体任务计划：`docs/exec-plans/`
- 原始参考资料、图片、PDF、截图：`docs/references/`
- Codex 使用说明：`instruction.md`

如果相关文档为空、缺失或内容不足，需要明确说明，不要假设它已经确定。

## 3. 文档职责

- `AGENTS.md`：只放协作规则和阅读路径。
- `docs/PRODUCT.md`：产品定位、用户、模块、边界、开放问题。
- `docs/STRUCTURE.md`：页面结构、导航关系、模块关系、用户路径。
- `docs/DESIGN.md`：长期设计原则、视觉语言、交互原则。
- `docs/ROADMAP.md`：阶段规划、优先级、暂缓事项。
- `docs/DECISIONS.md`：已确认且会影响后续工作的产品或技术决定。
- `docs/exec-plans/`：具体任务目标、范围、步骤、验证和结果。
- `docs/references/`：原始参考资料。参考资料是输入，不是最终决定。

不得使用 `AGENTS.md` 存放页面完整文案、任务进度、聊天记录、产品正文或修改流水。

## 4. Reference 生命周期与使用规则

- `docs/references/` 中的文件只是输入参考，不自动成为永久需求。
- 每个 Reference 应尽量标记状态：
  - `Active`：当前任务可使用；
  - `Consumed`：已使用并完成沉淀，后续默认不再使用；
  - `Archived`：仅供历史追溯。
- Agent 开始任务前，应先检查 Reference 的状态。
- 标记为 `Consumed` 或 `Archived` 的 Reference，不得默认参与新的产品、页面或视觉决策。
- 只有用户明确要求时，才可以重新使用已消费的 Reference。
- 使用一个 `Active` Reference 完成任务后，应检查是否需要：
  - 将长期设计规则沉淀到 `docs/DESIGN.md`；
  - 将页面和路由结构沉淀到 `docs/STRUCTURE.md`；
  - 将正式决定沉淀到 `docs/DECISIONS.md`；
  - 将实际完成结果记录到对应的 `docs/exec-results/`。
- 完成沉淀后，如该 Reference 属于一次性参考，应将其状态改为 `Consumed`。
- 不要为了保持 Reference 与当前页面一致而反复重写历史参考文件。
- 如果不确定某个 Reference 是否已经完成使用，不要自行标记，应向用户确认。

## 5. Markdown 语言规则

本项目所有 Markdown 文档优先使用中文。

允许保留英文的情况：

- 代码、变量名、命令、配置项；
- 第三方库、模型、工具和标准技术术语；
- 外部英文资料原文，但应尽量补充中文解释；
- 用户明确要求英文输出。

## 6. 范围控制

未经用户明确批准，不要直接执行：

- 初始化正式技术框架；
- 安装生产依赖；
- 创建数据库；
- 创建后台管理；
- 创建登录或权限系统；
- 创建爬虫、自动采集或第三方同步；
- 创建部署配置或执行部署；
- 增加未确认页面、模块或业务能力；
- 一次性迁移所有智能文档内容；
- 编造业务数据、员工数据、绩效结果、薪资数据、项目结果或用户转化数据。

对于合理但未批准的扩展，可以提出建议，但不得直接实施。

## 7. 必须先计划的任务

以下任务需要先检查仓库并制定计划，不应直接实现：

- 新模块建设；
- 技术栈选择或更换；
- 数据模型设计；
- 后台或数据库建设；
- 登录和权限系统；
- 自动报告生成；
- 爬虫或第三方数据同步；
- 大范围重构；
- 部署架构变化；
- 涉及敏感业务数据的功能。

复杂任务的计划放入 `docs/exec-plans/`。

## 8. 冲突处理

如果文档之间或用户新要求与已有文档冲突：

- 指出冲突位置；
- 说明冲突影响；
- 不静默选择其中一个版本；
- 等待用户确认后再修改已批准内容。

用户当前明确提出的新要求可能意味着已有文档需要更新，但不得在未说明冲突的情况下直接覆盖已批准内容。

每次修改已经确认的产品内容、页面结构或设计规则时，必须同步检查并更新所有直接相关的源文档、执行计划和阶段结果文件；如果不确定某个文档是否应修改，尤其是 `AGENTS.md` 或 `DECISIONS.md`，不得自行修改，应列出建议并等待用户确认。

## 9. 文件与命令安全

- 只在当前仓库中工作，除非用户明确授权访问其他位置。
- 不读取无关个人文件。
- 不修改项目外文件。
- 不写入密码、Token、API Key 或生产凭据。
- 不提交真实员工隐私、绩效或薪资数据。
- 不操作生产数据库或生产服务。
- 不执行 `git reset --hard`、`git clean -fd`、强制推送或重写历史，除非用户明确要求。
- 不覆盖用户未提交的修改。

涉及 `sudo`、`rm -rf`、全局安装、系统配置、Docker 数据卷、生产环境或项目外路径的操作，必须先说明原因和影响，并获得授权。

## 10. 验证与收尾

实现类任务完成后，根据项目已有工具执行必要检查。无法执行的检查必须说明原因。

最终回复应说明：

- 修改了什么；
- 没有做什么；
- 如何验证；
- 还剩哪些风险或待确认事项。

## 11. 修改本文件

只有长期稳定、会影响后续协作的规则才应写入本文件。

修改本文件前应说明：

- 要增加或修改什么；
- 为什么它属于长期规则；
- 会替代或影响哪些已有规则。
