# AGENTS.md

本文件只记录 agent 在本仓库中的长期协作规则。

## 1. 开始工作

- 先阅读 `README.md`，了解技术栈、命令和当前状态。
- 涉及已确认范围、技术或页面边界时，阅读 `docs/DECISIONS.md`。
- 涉及视觉、交互或中文阅读体验时，阅读 `docs/DESIGN.md`。
- 涉及优先级、阶段安排或未来能力时，阅读 `docs/ROADMAP.md`。
- 文档为空或内容不足时，应明确说明，不把未记录内容当作已确认结论。

## 2. 文档职责

- 各主要内容目录的 `README.md`：说明目录用途、文件职责、修改入口和安全边界；`src/pages/` 的说明写在 `src/README.md`，避免 Astro 生成额外路由。
- `docs/README.md`：核心文档导航及历史目录使用规则。
- `docs/DECISIONS.md`：已经确认的产品、结构和技术结论。
- `docs/DESIGN.md`：已经确认的长期视觉与交互规则。
- `docs/ROADMAP.md`：当前、下一步和暂缓事项。

不再为单个阶段常规创建产品定义、结构说明、执行计划、执行结果或 Reference 生命周期文档。复杂任务需要计划时，可以在开始执行前向用户展示计划；只有用户明确要求长期保存时，才写入仓库。

## 3. 工作原则

- 只做用户明确要求或已经批准的任务。
- 未确认内容标记为“假设”或“开放问题”。
- 不把总枢纽首页当作整个系统。
- 不为尚未确认的未来能力过度设计。
- 用户要求“只制定计划”时，完成计划后停止。
- 用户的新要求与 `docs/DECISIONS.md` 冲突时，指出冲突和影响，获得确认后再覆盖已批准结论。
- 修改已确认内容时，同步检查三份核心文档和 `README.md` 是否需要更新。

## 4. Markdown 语言

项目 Markdown 优先使用中文。代码、命令、配置、第三方产品名和标准技术术语可以保留英文。

## 5. 范围控制

未经用户明确批准，不要直接执行：

- 更换正式技术栈；
- 创建数据库、后台、登录或权限系统；
- 创建爬虫、自动采集或第三方同步；
- 创建部署配置或执行部署；
- 增加未确认页面、模块或业务能力；
- 编造业务数据、员工数据、绩效结果、薪资数据、项目结果或用户转化数据。

新模块、技术栈变化、数据模型、后台、权限、自动报告、外部同步、大范围重构、部署架构和敏感数据功能，必须先检查仓库并制定计划。

## 6. 文件与命令安全

- 只在当前仓库中工作，除非用户明确授权其他位置。
- 不修改或覆盖无关的未提交内容。
- 不写入密码、Token、API Key、生产凭据或真实员工隐私数据。
- 不操作生产数据库或生产服务。
- 不执行 `git reset --hard`、`git clean -fd`、强制推送或重写历史，除非用户明确要求。
- 涉及 `sudo`、`rm -rf`、全局安装、系统配置、Docker 数据卷、生产环境或项目外路径时，先说明原因和影响并获得授权。

## 7. 验证与收尾

实现完成后执行与风险相称的检查。无法执行的检查必须说明原因。

最终回复应说明：

- 修改了什么；
- 没有做什么；
- 如何验证；
- 仍有哪些风险或待确认事项。

## 8. 修改本文件

只有长期稳定、会影响后续协作的规则才写入本文件。修改前应说明新增或替换的规则及其影响。
